## 1. Introduction

The Low Latency Communication Engine (LLCE) is an autonomous accelerator designed to offload communication tasks on the CAN, LIN and Flexray protocols. It is a self-contained module, separate from the application cores that would run applications (host cores). Therefore, it has some additional setup prerequisites.

LLCE is a flexible communication engine to support traditional network protocols:

- 16 CAN interfaces supporting CAN-FD
- 4 LIN interfaces
- 1 FlexRay interface

LLCE accelerator contains 4 execution cores (ARM M0+ cores) with targeted functionalities. Each core runs its own application in the firmware.

## 1.1. LLCE Firmware

The LLCE runs its own software. This is provided in two variants  - Standard  (binary)  and  FDK (Firmware Development Kit). The FDK provides more functionality, and comes with the source code, allowing for customization and extension of the code run by LLCE.

For either variant, the binary distribution consists of two sets of 4 files (one for each of the ARM M0+ cores). Each of the 4 files in a set is to be loaded on the corresponding core of the LLCE.

The first set consists of ELF files, which may be loaded using code-loading tools.

The second set consists of pseudo-binary files. These contain just the executable code extracted from the above ELFs, transformed into C arrays. This allows easy incorporation into the host application, when this is desired.

The firmware package also includes a Firmware User Guide, and the LLCE host interface headers. The Firmware User Guide provides more in-depth information on the operation of LLCE and the LLCE Interface can be used to communicate with the LLCE accelerator by customers who do not want to use the Autosar drivers.

## 1.2. LLCE Autosar drivers (optional, recommended)

To ease application development, a set of Autosar compatible drivers are provided in the same package as the firmware. These drivers are designed to integrate with the NXP RTD suite and resemble to a degree the corresponding RTD drivers.

For customers using the LLCE Standard package, the following drivers are provided:

## 2. LLCE Startup

The LLCE has its own clock, SRAM area, and code areas. All these must be initialized, in this order, before an application can start using LLCE.

Until all components have been prepared, the LLCE system shall be kept in reset by using the System reset control register (see chapter Low Latency Communication Engine (LLCE) in the S32G Reference Manual [2]).

## 2.1. LLCE Clocking

The LLCE has a gated clock. Before any operation may be done in the LLCE memory area,

LLCE\_SYS\_CLK must be configured to a suitable value (see Chapter Clocking in the S32G Reference Manual [2]) and MC\_RGM Reset Domain 3 and MC\_ME Partition 3 (see Chapter Reset and Chapter Mode Entry Module in the S32G Reference Manual [2]) must be configured accordingly.

In addition, each of the communication protocols supported by the LLCE has a separate clock, derived from the main chip clocks. See section LLCE subsystem clocking overview of the S32G Reference Manual [2].

The required settings may be done using the MCAL MCU module or direct writing to the relevant registers. Attempting to access any address in the LLCE memory space before enabling the module may result in hard faults.

## 2.2. LLCE SRAM

The  LLCE  module  occupies  the  memory  space  0x43000000  -  0x43FFC000.  This  area  includes hardware modules, code RAM from which the LLCE Firmware is executed, and general-purpose RAM. The  LLCE  Firmware  will  initialize  all  LLCE-internal  hardware  modules,  but  ECC  SRAM  must  be initialized before the firmware can be loaded and run.

The following areas are ECC SRAM and must be initialized with valid 32-bit aligned writes before they may be used:

- 0x43000000 - DTE\_END\_SEC : DTE code section

- 0x43100000 - PPE\_RX\_END\_SEC : PPE\_RX code section

- 0x43200000 - PPE\_TX\_END\_SEC : PPE\_TX code section

- 0x43300000 - FRPE\_END\_SEC : FRPE code section

- 0x43800000 - 0x4384FFFF: shared between LLCE cores, and host

| Platform   | DTE_END_SEC   | PPE_RX_END_SEC   | PPE_TX_END_SEC   | FRPE_END_SEC   |
|------------|---------------|------------------|------------------|----------------|
| S32G2      | 0x43001FFF    | 0x43107FFF       | 0x43207FFF       | 0x4330FFFF     |

Memory  initialization  may  be  done  using  classical  writes  from  code,  memory  controller,  DMA transfers, or any other memory writing method. Using 0 for the initial write is recommended. Care must be taken to not write to areas other than mentioned above, which may lead to hard faults or unpredictable behavior. Initialization may also be combined with code loading, as described in the next section.

## 2.3. LLCE Firmware Loading

LLCE Firmware must be loaded on each core's instruction memory area. This may be done through any memory transfer method, such as DMA. ELF files may also be loaded directly, provided the previous steps regarding LLCE clocking and memory initialization have been followed.

To optimize startup times, firmware loading may be combined with memory initialization. If using a transfer method that guarantees 32-bit aligned writes, there is no need to do an initialization pass before. However, the portion of each RAM section that is not filled with code must still be initialized. The shared memory area must be initialized as well. The sample applications provided with  the LLCE firmware releases include a firmware loading module to demonstrate this technique using simple write loops.

After firmware loading is done, the System reset control register must be written to enable the cores (get the cores out of reset) one at a time, or all at once. The sample applications provided with the LLCE firmware releases include an example for setting this.

## 2.4. LLCE Startup reporting

The LLCE shared memory has a reserved area to indicate successful startup and other useful diagnostic information. It can be checked to diagnose possible startup issues with LLCE. See section Base addresses of the shared memory areas in the LLCE Firmware User Guide [1] for information on its location and information encoded.

- the configuration specifies 8 controllers assigned to host1 and other 8 assigned to host2. They are sending  frames  simultaneous  through  LLCE.  In  the  last  part,  host1  sends  frames  to  host2  by internal routing method. Also, the demo generates 2 elf files, corresponding to every host, which takes half of the SRAM each one for the code area.

## Other useful details:

- A solution for loading the firmware and enabling the ECC RAM is provided in firmware\_loading folder, by copying raw bytes from host memory to LLCE internal memory and enable LLCE cores in order to start running. For a better understanding, follow up the comments in Llce\_Firmware\_Load() function in Llce\_Firmware\_Load.c source file.
- Multiple baudrate configurations for this sample application are available and can be referenced in Elektrobit Tresos, in each CanController configuration.
- o Default baudrate chosen in every demo has index 1 in Tresos and corresponds to a baudrate of 500 Kbps in Nominal Phase and 2 Mbps in FD Phase.
- o The other configs are (index 0: 250Kbps, 500 Kbps) and (index 2: 1Mbps, 4Mbps).
- o When configuring another baudrates, be aware and verify all warnings raised by Elektrobit Tresos regarding baudrate configuration.
- The CAN driver does the timeout detection (seconds) for non-blocking functions using the Autosar OS APIs OsIf\_GetCounterValue and OsIf\_GetElapsedValue.
- Clock tree initialization and LLCE partition activation can be done either using MCU driver, which is the default in demos using RTD drivers, or using static clock initialization (sys\_init).

## 6. Frequently Asked Questions

## 6.1. Errors while loading the application or gigantic binary file

If an application incorporating LLCE results in a huge (100s of MB or even GB) memory image, or the loader causes an error such as a hard fault, check the linker file.

The Autosar drivers require a memory area to be defined in the LLCE shared memory described in section 2.2 LLCE SRAM , and this is recommended for direct host interface applications as well. However, this section is meant to be symbolic and serve for variable placement only. It should never be initialized or accessed before the LLCE Init is performed. The issue may be caused by the linker or loader trying to do this.
